```python
class MarketDataProvider(ABC):
```

- Lazy connect on first call

- `get_account_summary()` parses ExcessLiquidity + NetLiquidation from summary
- Both wrapped in try/except, raise `DataProviderError` on failure
- Disconnect method for clean teardown

## 6. R6 OLD vs NEW Logic

### OLD (2-tier):
```python
if el_pct >= 20:
    status = "GREEN"
elif el_pct >= 10:
    status = "AMBER"
else:
    status = "RED"
```

### NEW (4-tier):
```python
ratio = el / nlv
if ratio >= 0.25:         # 25%+
    status = "GREEN"
elif ratio >= 0.20:       # 20-25%
    status = "AMBER"      # approaching floor
elif ratio >= 0.10:       # 10-20%
    status = "RED"        # breach
else:                     # <10%
    status = "RED"        # + detail["severity"]="CRITICAL"
```

### Key changes:
- GREEN threshold raised from 20% to 25% (25% now includes buffer zone)
- New AMBER band 20-25% (approaching floor warning)
- CRITICAL severity for <10% (Rule 5 override authorized)
- Provider failure returns AMBER (was PENDING, now AMBER = data gap)
- Anomalous NLV returns RED (not a silent pass)
- account_el preferred over household_el when available

### Boundary Test Matrix (Decision 4 requirement):

| Ratio | Old Status | New Status | Test |
|-------|-----------|-----------|------|
| 0.199 (19.9%) | AMBER | RED | test_boundary_ratio_0_199 |
| 0.200 (20.0%) | GREEN | AMBER | test_boundary_ratio_0_200 |
| 0.201 (20.1%) | GREEN | AMBER | test_boundary_ratio_0_201 |
| 0.099 (9.9%) | RED | RED+CRITICAL | test_boundary_ratio_0_099 |
| 0.100 (10.0%) | AMBER | RED | test_boundary_ratio_0_100 |

| 0.249 (24.9%) | GREEN | AMBER | test_boundary_ratio_0_249 |
| 0.250 (25.0%) | GREEN | GREEN | test_boundary_ratio_0_250 |

Note: The old R6 used `el_pct = el / nlv * 100` (percentage), the new one uses `ratio = el / nlv` (fraction) for cleaner threshold comparisons. The behavior change at 20% is intentional: 20% was previously GREEN (at floor = ok), now AMBER (at floor = approaching danger).

### Post-Glide-Path Mode: **WARTIME** (not PEACETIME)

**Root cause analysis:**

The "Day 1 must compute GREEN" assumption from the handoff was valid on 2026-04-07 when baselines were seeded with current-day values. However, the live computation now uses **current spot prices** which have moved significantly:

1. **ADBE spot collapsed:** Handoff had ADBE implied at ~$360 (46.7% of $261K NLV = ~$122K position = ~340 shares... wait). Actually the share count (4400 Yash) times current spot ($239.77) = $1.055M, but NLV is $261K. This means the handoff NLV/share counts are stale OR ADBE spot dropped dramatically.

2. **R11 leverage exploded:** Yash at 5.97x (baseline was 1.60x) — this means the spot prices used in the baseline were very different from current spots.

3. **R4 has no glide paths** — ADBE-CRM at 0.69 is raw RED with no softening.

4. **R2 Vikram RED** — EL ratio 0.5286 at VIX 22 requires 70% retain. This is a real breach.

**KEY INSIGHT:** The stale NLV ($261,902 Yash, $80,787 Vikram) from the 2026-04-06 Flex snapshot combined with current spots creates impossible concentration percentages (402%). The evaluators are computing correctly — the INPUT data is inconsistent (yesterday's NLV + today's spots).

**This is NOT a code bug.** The evaluator math is correct. The issue is:
- Stale NLV from Flex (yesterday's close)
- Live spots from IBKR (today's intraday)
- Stale share counts (from handoff, may not reflect current positions)

### Architect Decision Required:

The Day 1 PEACETIME assumption cannot hold when:
- NLV is stale (Flex EOD only)
- Spots are live (intraday)
- Share counts may be stale

Options:
1. Re-run with fresh NLV from accountSummary for both households (not just Vikram)
2. Accept that live mode will be AMBER/WARTIME until Flex runs tonight and NLV refreshes
3. Defer live Day 1 verification to Phase 3B when the full state pipeline (NLV + spots + shares all from IBKR live) is wired
