## 4. Strategy Composition Design

### 4a. Atomic Strategies (in `tests/property/strategies.py`)

```
TICKERS = ['TEST', 'AAPL', 'MSFT']  # small set for shrinking
ACCOUNTS = ['U21971297', 'U22076329']  # both Yash_Household
HOUSEHOLD = 'Yash_Household'
DATES = sorted date strings 20260101..20260131
STRIKES = floats 50..500, step 5
QUANTITIES = integers 1..10 (shares in 100-lots for options → 100..1000 for stock)
PRICES = floats 0.10..50.00
```

**`trade_event_st()`** — base strategy, draws a single `TradeEvent` with constrained fields:
- ticker, household_id, account_id fixed per composite call (ensures I3/I15/I16 compliance)
- date_time from sorted date pool
- transaction_type ∈ {ExchTrade, BookTrade}
- asset_category ∈ {STK, OPT}
- buy_sell ∈ {BUY, SELL}
- open_close ∈ {O, C}
- Unique transaction_id per draw

**`csp_open_close_pair_st()`** — composite: generates a matched CSP_OPEN + CSP_CLOSE pair (same strike/expiry/right=P, O then C, sequential dates).

**`assignment_pair_st()`** — composite: CSP_OPEN → ASSIGN_OPT_LEG + ASSIGN_STK_LEG (same day, strike-matched).

**`transfer_pair_st()`** — composite: TRANSFER_OUT from account A + TRANSFER_IN to account B (same qty, same ticker, same household).

**`excluded_ticker_events_st()`** — generates events with ticker ∈ excluded set.

**`satellite_long_opt_st()`** — LONG_OPT_OPEN + optional LONG_OPT_CLOSE sequence.

**`valid_event_sequence_st()`** — higher-order composite: draws N events from a mix of the above, sorts by canonical_sort_key, ensures all share same (household, ticker). This is the workhorse strategy for most properties.

### 4b. Key Design Decisions

1. **Constrain rather than filter.** Generate valid events directly rather than drawing random fields and filtering. Hypothesis `assume()` wastes examples.

2. **Single ticker + single household per test.** Walker requires homogeneous input. Generate one ticker/household, vary everything else.

3. **Date ordering matters.** Events within a day can be shuffled (for ordering-independence test), but days must be chronological. Strategy generates dates from a sorted pool.

4. **Quantities always positive.** TradeEvent.quantity is always positive per spec. buy_sell determines direction.

5. **transaction_id uniqueness.** Use `st.uuids()` or counter to guarantee unique IDs.

### 4c. Shrinking Considerations

Hypothesis auto-shrinks, but we should seed `@example` for known bug patterns:
- **ADBE assignment chain:** 2 puts assigned same day at different strikes → strike-match regression
- **PYPL roll sequence:** close + reopen same day → premature closure risk
- **Vikram transfer:** intra-household transfer with basis preservation
- No real bugs found during W3.1-W3.5 that weren't immediately fixed, but the assignment-strike-match pattern is the most subtle

## 8. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Strategy generates invalid event combos that crash walker | False bug reports | Constrain strategies to generate semantically valid events; use `assume()` sparingly |
| EXPIRE_WORTHLESS reverse-scan heuristic has edge cases | May find real bugs | This is the POINT — report, don't fix |
| Property test finds assignment basis non-determinism | Real bug if true | Report immediately, don't weaken test |
| Runtime exceeds 60s | CI friction | Tune max_examples down to 50 if needed |
| `canonical_sort_key` has undocumented tie-breaking | Ordering test may fail | Read sort key implementation, match in strategy |
