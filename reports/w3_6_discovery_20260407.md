## 3. `/reconcile` Handler

**Location:** `telegram_bot.py:9481-9694`

**Current flow:**
1. Load all events, group by (household, ticker), walk_cycles per group
2. Count `UnknownEventError` as `frozen_count` (caught, not surfaced)
3. Cross-checks A (realized P&L), B (cost basis), C (NAV recon)
4. R5 order health (T+2 orphans, stale staged)
5. Output as `<pre>` HTML block

**Current output format (lines 9628-9690):**
```
RECONCILIATION REPORT
Last sync: #42 2026-04-07T18:30:00 (OK, 150 rows)
Parity: 438/438
Cycles: 86 (14 active, 84 wheel, 2 satellite)
Frozen: 0

A (realized P&L): 49/49
B (cost basis): 14/14
C (NAV recon): 3/4
  U21971297: $0.15
  U22388499: -$21.53
  U22076329: $0.00
  U22076184: $0.00

ORDER HEALTH: clean
```

**Note:** `get_walker_warnings()` is NOT called in `/reconcile` today. Also, the current loop catches `UnknownEventError` but discards any warnings accumulated before the raise — `walk_cycles()` clears `_walker_warnings` on each call, so if group N raises, warnings from group N are lost, and group N+1 starts fresh. This is a **stale-warnings bug**: `get_walker_warnings()` only reflects the LAST group's walk, not all groups. Needs accumulation across groups.

## 5. Proposed Design

### 5a. Telegram `/reconcile` — append after ORDER HEALTH section

Accumulate warnings across ALL (household, ticker) groups, then append:

```
WALKER WARNINGS: 0
```

Or if warnings exist:
```
WALKER WARNINGS: 3
  [COUNTER_GUARD] AAPL Yash_Household: Non-negative guard: shares_held would go to -100 (current=0, delta=100) on AAPL 2026-04-07 tid=12345
  [ORPHAN_TRANSFER] MSFT Yash_Household: Orphan TRANSFER_IN for MSFT on 2026-04-01 (acct=U21971297, tid=67890). Position not attached to any cycle.
  [EXCLUDED_SKIP] SPX: Excluded ticker skipped (3 events)
```

Truncation: if >10 warnings, show first 10 + `... and N more`.

**Try/except wrapper:** If `get_walker_warnings()` raises, show `WALKER WARNINGS: ⚠️ walker unavailable`.

### 5b. Command Deck badge — new cell between Sector and Sync

```html
<div class="flex items-center gap-1.5 text-xs"
     title="Walker warnings from last reconciliation walk">
  <span class="text-slate-500">Warn</span>
  {% if top.walker_warnings is not none %}
  <span class="num font-semibold {% if top.walker_warning_count > 0 %}text-amber-400{% else %}text-emerald-400{% endif %}">
    {{ top.walker_warning_count }}
  </span>
  {% else %}
  <span class="text-slate-500">—</span>
  {% endif %}
</div>
```

Color: 0 = emerald, >0 = amber. No rose — warnings are not errors.

**Route change:** `build_top_strip()` will call `get_walker_warnings()` inside try/except. Needs to run a full walker walk first (same as `/reconcile` does) — OR read from a persisted `walker_warnings_log` table.

**Key design question:** Running a full walker walk on every deck page load is expensive (~0.5s). Two options:
- **Option A:** Persist warnings to `walker_warnings_log` table during flex_sync / reconcile, read from table in deck. (Spec item d asks for this table.)
- **Option B:** Cache warnings in memory after `/reconcile` run, show stale count in deck with timestamp.

**Recommendation:** Option A (table). It's in scope per spec, and the Sync cell already reads `master_log_sync` for last sync time — same pattern.

### 5c. `walker_warnings_log` table schema (spec item d)

```sql
CREATE TABLE IF NOT EXISTS walker_warnings_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    sync_id     INTEGER,              -- FK to master_log_sync, NULL if from /reconcile
    timestamp   TEXT NOT NULL DEFAULT (datetime('now')),
    ticker      TEXT,
    household   TEXT,
    event_type  TEXT,                  -- COUNTER_GUARD, UNKNOWN_ACCT, ORPHAN_TRANSFER, EXCLUDED_SKIP
    reason      TEXT NOT NULL,         -- full warning string
    event_payload TEXT                 -- JSON of the triggering event (optional)
);
```

**Write locations:** Only `/reconcile` handler and `flex_sync.py` post-walk.
**Read locations:** Command Deck `build_top_strip()` reads latest sync_id's warnings.

## 6. Stale-Warnings Bug

**Critical finding:** `walk_cycles()` clears `_walker_warnings` on entry (line 570-571). In `/reconcile`'s loop (line 9511-9515), each `_wc(list(grp))` call clears the previous group's warnings. After the loop, `get_walker_warnings()` returns only the LAST group's warnings.

**Fix required:** Accumulate in the caller:
```python
all_warnings = []
for (hh, tk), grp in _gb(combined, key=lambda e: (e.household_id, e.ticker)):
    try:
        all_cycles.extend(_wc(list(grp)))
        all_warnings.extend(get_walker_warnings())  # capture before next call clears
    except _UE:
        frozen_count += 1
```

This is a pre-existing bug unrelated to W3.6, but W3.6 must fix it to surface warnings correctly.

## 7. Edge Cases

| Case | Handling |
|------|----------|
| 0 warnings (current state) | Show `WALKER WARNINGS: 0` / badge emerald `0` |
| >10 warnings in Telegram | Truncate to 10 + `... and N more` |
| >20 warnings | Same truncation. Badge shows count. Click-to-modal on deck shows all. |
| `get_walker_warnings()` raises | `WALKER WARNINGS: ⚠️ walker unavailable` / badge shows `—` |
| `UnknownEventError` in walk loop | Already caught as frozen_count. Warnings from that group are lost (acceptable — the freeze IS the signal). |
| Empty warnings list + frozen > 0 | Show both: `Frozen: 2` already exists, `WALKER WARNINGS: 0` is accurate (warnings are separate from frozen). |
| Severity color in deck | No severity in current schema. All warnings = amber. If severity is added later, rose = error, amber = warn. |
